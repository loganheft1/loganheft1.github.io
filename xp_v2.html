What I Own

As Technical PM for Consumer & Business Credit Cards, I own the data infrastructure and decision engine that determines credit line changes for millions of customers. This means the pipelines, the vendor integrations, the model inputs, the validation layers, and the policies that govern when decisions can and cannot run.

The role sits at an unusual intersection: I need to be technical enough to write API contracts and diagnose production incidents, while also being commercial enough to quantify risk-adjusted return and navigate Fair Lending compliance.

AI Income Encoder — The Problem

Our proactive credit line increase campaigns excluded a meaningful segment of creditworthy customers — those without directly verifiable income through standard documentation. These customers had strong credit profiles but fell outside the decisioning logic, meaning we weren't serving them and they weren't getting increases they warranted.

I identified this gap through data analysis: a segment with favorable credit signals that was systematically being dropped from campaigns due to a single missing field.
The Solution I Built

Championed an LLM-based income derivation model — the AI Encoder — that infers income from transactional behavior and other available signals where direct verification isn't possible.

  • Owned the full roadmap: concept → data science scope → model validation → Fair Lending review → API integration → deployment
  • Coordinated model validation through the credit risk governance process, including disparate impact testing against ECOA-protected classes
  • Extended the decisioning engine payload schema to accept model-generated income fields with confidence score gating
  • Authored the integration spec, test plan, and monitoring runbook

Batch → Real-Time: Why It Mattered

The legacy decisioning architecture ran overnight batch jobs — pulling data, running decisions, writing outcomes, all on a fixed schedule. This meant customers requesting an instant credit line increase were getting a decision rendered on data that was potentially 24+ hours old. Worse, it blocked four critical capabilities that required real-time signal freshness.

What I Delivered
  • Instant CRCLI — Customer-requested credit line increases, now with sub-5s response targets
  • FICO 10T Integration — Next-generation FICO scoring model with trended credit data
  • FEMA API — ZIP-code level disaster exclusions replacing error-prone state-level manual process
  • Automated Income Verification — Real-time income check replacing batch document review
Authored all API contracts, ran shadow-mode testing (parallel batch + real-time) before every cutover, and managed 25+ engineers across 4 migration tracks simultaneously.
Technical Complexity

The hardest part wasn't the APIs — it was the audit trail. Batch decisions logged themselves naturally at the end of a cycle. Real-time decisions needed event-based logging that satisfied compliance, could be reconstructed for regulatory disclosure, and could be queried by analytics without degrading response latency. I designed the event schema and worked with engineering to implement atomic logging before any decision response returned.

I also built circuit breaker logic into all vendor integrations: if a vendor API timeout rate exceeded 15%, we automatically rerouted to a backup endpoint rather than failing the decision.

The Incident That Changed How We Work

An upstream table that populated our student segmentation flag failed to load before a decisioning cycle. The engine saw a null flag, treated every student as a non-student, and applied the standard rule set — which is harsher for students, a segment we intentionally handle carefully.

We decreased credit lines for customers we shouldn't have touched. The remediation required outreach to affected customers, decision reversals, and reconciliation across systems. It was expensive, damaging to customer relationships, and entirely avoidable.

The Floodgate I Built

I designed a pre-campaign check — a "floodgate" — that queries all upstream tables before any decisioning cycle begins. Each critical field has a registered check with configurable threshold logic. If anything fails, the campaign doesn't run. Full stop.

  • Null / completeness checks per field
  • Statistical drift detection (z-score variance from 90-day baseline)
  • Range validation and semantic integrity checks
The drift check caught a second incident before it became a crisis: an upstream provider changed a field from monthly to annual income with no notification. Our engine would have given every customer 12× their actual income. The z-score on that field was off the charts. We caught it before a single decision ran.
"Bad data that looks clean is the most dangerous kind. A null field is easy to catch. A field that passes all completeness checks but has silently changed its semantic meaning will fool you every time."
WSJF Prioritization — Three P1s, One Team

At one point I had three simultaneous Priority 1 requests from different stakeholders: a FEMA API integration, an email delivery channel integration, and a waterfall decisioning logging overhaul. Each team believed their ask was the most urgent. I needed a defensible framework — not just my gut.

The waterfall logging eventually got a lightweight dbt-based solution — 80% of the value at 20% of the effort. WSJF taught me to always ask: is there a simpler path to 80% of the value?
LLM Transaction Taxonomy

Beyond infrastructure, I identified an opportunity to enrich the decisioning model with new signals. Our existing model relied heavily on bureau and FICO inputs — lagging indicators. I worked with data science to scope an LLM-based transaction classifier that generates a normalized category taxonomy from raw transaction descriptions.

The result: 400+ new leading credit health signals, feeding into a feature store that the model reads at execution time. This improved model precision across the full credit card portfolio without touching the core decisioning logic.

The Incident: 25% Failure Rate, Live Campaign

During a proactive credit line increase campaign — generating approximately $40 in Risk-Adjusted Return per account — our Decision Modeler began experiencing a surge in processing failures affecting hundreds of thousands of accounts. The immediate team reaction was to shut down the campaign.

I argued to stay live with enhanced monitoring, and I was right. Here's how I reasoned through it.

The Diagnosis

I convened an emergency war room with the line-of-business Risk lead, the Credit Card Director, and tech leadership. While the team pulled logs, I ran two parallel analyses:

  • Root cause: 25% of requests were timing out at the vendor API layer. Infrastructure tracing showed the vendor's load balancer was failing intermittently — not the decision logic itself.
  • Disparate impact check: I immediately ran demographic analysis on failed vs. successful accounts. No correlation with credit score, geography, or protected class proxies. The failures were statistically random.
This distinction — random infrastructure failure vs. systematic logic failure — was the linchpin. A systematic logic failure requires immediate shutdown. A random infrastructure failure can be monitored through.
The Decision Framework

I presented three data points to leadership: 75% of decisions processing successfully, stochastic error pattern with no demographic skew, and we were still within the pre-notification window meaning no customers had been informed of any decisions yet.

Proposal: monitored-live approach — stay operational with hourly error-rate tracking and an automatic kill switch if failures exceeded 30%. FICO resolved their infrastructure issue within 8 hours.
